---
title: "Quick start"
description: "Go from zero to a working OMX session in four steps."
---

<Steps>
  <Step title="Install">
    Install both the Codex CLI and OMX globally with npm:

    ```bash
    npm install -g @openai/codex oh-my-codex
    ```

    <Note>
      Node.js 20 or later is required. Run `node --version` to confirm before installing.
    </Note>
  </Step>

  <Step title="Set up">
    Run `omx setup` inside your project directory. This installs skills, agent prompts, MCP server config, and an `AGENTS.md` scaffolding:

    ```bash
    omx setup
    ```

    Setup will ask whether to install at **user scope** (files go to `~/.codex`) or **project scope** (files go to `./.codex`). User scope is the default and works across all your projects.

    Then verify the installation is healthy:

    ```bash
    omx doctor
    ```

    A passing run looks like:

    ```text
    oh-my-codex doctor
    ==================

      [OK] Codex CLI: installed
      [OK] Node.js: v20.x.x
      [OK] Codex home: /Users/you/.codex
      [OK] Config: config.toml has OMX entries
      [OK] Prompts: N agent prompts installed
      [OK] Skills: N skills installed
      [OK] AGENTS.md: found in /Users/you/.codex/AGENTS.md
      [OK] State dir: /your/project/.omx/state
      [OK] MCP Servers: N servers configured (OMX present)

    Results: 9 passed, 0 warnings, 0 failed

    All checks passed! oh-my-codex is ready.
    ```

    If any checks fail, re-run `omx setup` and then `omx doctor` again.
  </Step>

  <Step title="Launch">
    Start a Codex session with the recommended OMX flags:

    ```bash
    omx --madmax --high
    ```

    - `--madmax` bypasses Codex approval prompts and sandbox restrictions
    - `--high` sets the model reasoning effort to `high` for better planning and code quality

    <Warning>
      `--madmax` disables Codex's approval sandbox. Only use it in environments where you trust the agent to execute freely. Omit the flag to keep interactive approval prompts.
    </Warning>

    For a safer or lower-resource launch:

    ```bash
    omx          # standard launch, approval prompts enabled
    omx --high   # high reasoning, approval prompts still on
    ```
  </Step>

  <Step title="Run your first workflow">
    Once inside the Codex session, use the canonical OMX workflow. Start with clarification when the request or boundaries are still unclear:

    ```text
    $deep-interview "clarify the authentication change"
    ```

    When the scope is clear, approve a plan:

    ```text
    $ralplan "approve the auth plan and review tradeoffs"
    ```

    Then choose your execution style:

    <CodeGroup>
      ```text Parallel team execution
      $team 3:executor "execute the approved plan in parallel"
      ```

      ```text Single-owner persistent loop
      $ralph "carry the approved plan to completion"
      ```
    </CodeGroup>

    Use `$team` when the work can be split across multiple parallel lanes. Use `$ralph` when you want one agent to own and verify the entire task through to completion.
  </Step>
</Steps>

## In-session surfaces

| Surface | Use it for |
| --- | --- |
| `$deep-interview "..."` | Clarifying intent, boundaries, and non-goals |
| `$ralplan "..."` | Approving the implementation plan and tradeoffs |
| `$ralph "..."` | Persistent completion and verification loops |
| `$team "..."` | Coordinated parallel execution |
| `/skills` | Browsing all installed skills and helpers |

## Troubleshooting

<Accordion title="`omx` command not found after install">
  The npm global bin directory may not be in your `PATH`. Find it with:

  ```bash
  npm config get prefix
  ```

  Add `<prefix>/bin` to your shell's `PATH` in `~/.bashrc`, `~/.zshrc`, or equivalent, then restart your shell. Re-run the install:

  ```bash
  npm install -g @openai/codex oh-my-codex
  ```
</Accordion>

<Accordion title="Prompts not available inside Codex">
  Skills and prompts are installed to `~/.codex/prompts/` and `~/.codex/skills/` by default. Confirm the files are there:

  ```bash
  ls ~/.codex/prompts/
  ls ~/.codex/skills/
  ```

  If the directories are missing or empty, re-run setup:

  ```bash
  omx setup --force
  ```
</Accordion>

<Accordion title="`omx doctor` reports config issues">
  Run doctor to see exactly which checks fail:

  ```bash
  omx doctor
  ```

  Most config issues are fixed by running:

  ```bash
  omx setup --force
  ```

  If `config.toml` has a duplicate `[tui]` section (possible after a version upgrade), OMX will repair it automatically on the next `omx` launch.
</Accordion>

<Accordion title="Skills not loading inside Codex">
  Verify the skill files were installed. Each skill should have its own directory with a `SKILL.md` file:

  ```bash
  ls ~/.codex/skills/
  ```

  If they are missing, re-run `omx setup`. On project scope, check `.codex/skills/` in your project root instead.
</Accordion>

<Accordion title="High CPU on Intel Mac at startup">
  On some Intel Macs, launching with `--madmax --high` can spike `syspolicyd` and `trustd` CPU while macOS Gatekeeper validates concurrent process launches. To mitigate:

  ```bash
  xattr -dr com.apple.quarantine $(which omx)
  ```

  You can also add your terminal app to the Developer Tools allowlist in macOS Security settings, or use a lower-concurrency launch by omitting `--madmax --high`.
</Accordion>
