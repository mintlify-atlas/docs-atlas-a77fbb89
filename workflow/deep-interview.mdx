---
title: "$deep-interview"
description: "Socratic clarification workflow that gates execution until intent, scope, and non-goals are explicit."
---

`$deep-interview` is the first step in the canonical OMX workflow. It runs a structured Socratic interview before any planning or implementation begins, turning vague requests into execution-ready specifications. Execution is gated until ambiguity drops below the profile threshold.

## What it does

The skill opens an intent-first interview loop. It asks one focused question per round, targets the weakest clarity dimension, scores ambiguity after every answer, and will not hand off to execution while non-goals or decision boundaries remain unresolved.

On completion it writes two artifacts:

- **Transcript** — `.omx/interviews/{slug}-{timestamp}.md`
- **Clarified spec** — `.omx/specs/deep-interview-{slug}.md`

The spec becomes the requirements source of truth for `$ralplan`, `$ralph`, and `$team`.

## When to invoke

- The request is broad, ambiguous, or missing concrete acceptance criteria
- The user says "interview me", "don't assume", or "ouroboros"
- You need a requirements artifact before planning or implementation starts
- Scope or non-goals are still undefined

<Note>
Skip `$deep-interview` only when the task already names specific files or symbols and has explicit acceptance criteria. Vague prompts like "improve the app" should always go through this step first.
</Note>

## How to use

```text
$deep-interview "clarify the authentication change"
```

### Depth profiles

Three profiles control how rigorously the interview explores the problem:

<CodeGroup>

```text Quick (--quick)
$deep-interview --quick "add a retry policy"
```

```text Standard (default)
$deep-interview "redesign the auth flow"
```

```text Deep (--deep)
$deep-interview --deep "migrate the database layer"
```

</CodeGroup>

| Profile | Flag | Ambiguity threshold | Max rounds |
|---------|------|---------------------|------------|
| Quick | `--quick` | ≤ 0.30 | 5 |
| Standard | *(default)* | ≤ 0.20 | 12 |
| Deep | `--deep` | ≤ 0.15 | 20 |

## Interview loop behavior

Each round follows this sequence:

<Steps>
  <Step title="Preflight context intake">
    OMX derives a task slug, checks for an existing context snapshot under `.omx/context/`, and creates one if none exists. This snapshot carries the task statement, probable intent, known facts, constraints, and open questions.
  </Step>
  <Step title="Generate the next question">
    The skill targets the weakest clarity dimension in the current stage. Stages run in priority order: intent and scope first, then feasibility, then brownfield grounding (when working in an existing codebase).
  </Step>
  <Step title="Score ambiguity">
    After every answer the skill scores each dimension and recomputes the weighted ambiguity score. Scores and progress are shown transparently each round.
  </Step>
  <Step title="Pressure pass">
    Before crystallizing, the skill revisits at least one prior answer with an evidence-focused, assumption-focused, or tradeoff-focused follow-up. Rotating to a new dimension before deepening the current one is not permitted.
  </Step>
  <Step title="Crystallize and hand off">
    When ambiguity is below the threshold and non-goals and decision boundaries are both explicit, the skill writes the transcript and spec artifacts and presents handoff options.
  </Step>
</Steps>

## Lock behavior

While `$deep-interview` is active, the session is in requirements mode. The skill does not implement code, does not call execution tools, and does not hand off until the readiness gates are satisfied:

- Non-goals must be explicit
- Decision boundaries must be explicit
- At least one assumption pressure pass must be complete

<Warning>
If the interview ends via early exit or hits the maximum round cap, the residual-risk state is preserved in the handoff so the downstream skill knows it inherited a partially clarified brief.
</Warning>

## What it produces

The clarified spec at `.omx/specs/deep-interview-{slug}.md` includes:

- Metadata (profile, rounds, final ambiguity score, context type)
- Intent — why the user wants this change
- Desired outcome
- In-scope and out-of-scope items
- Decision boundaries — what OMX may decide without confirmation
- Constraints
- Testable acceptance criteria
- Exposed assumptions and their resolutions
- Pressure-pass findings

## Handoff to $ralplan

After the spec is written, `$deep-interview` presents handoff options. The recommended next step is always `$ralplan`:

```text
$ralplan "approve the auth plan and review tradeoffs"
```

`$ralplan` treats the deep-interview spec as the requirements source of truth. It does not repeat the interview; it refines architecture and feasibility around the clarified intent and boundaries.

<Tip>
You can also hand off directly to `$ralph` or `$team` when the clarified spec is strong enough for execution without an additional consensus gate.
</Tip>

## Resume

If the session is interrupted, rerun `$deep-interview`. The skill reads persisted mode state via `state_read(mode="deep-interview")` and resumes from where it left off.

## Advanced options

<AccordionGroup>
  <Accordion title="--autoresearch flag">
    Switches the interview into autoresearch-intake mode, focused on producing launch-ready artifacts for `omx autoresearch`. The interview converges on a research mission and writes canonical artifacts under `.omx/specs/autoresearch-{slug}/`.
  </Accordion>
  <Accordion title="Challenge modes">
    The skill activates assumption stress-test modes at defined thresholds: **Contrarian** (challenges core assumptions), **Simplifier** (probes minimal viable scope), and **Ontologist** (asks for essence-level reframing when the user keeps describing symptoms). Each mode runs once per session.
  </Accordion>
  <Accordion title="Brownfield vs greenfield scoring">
    Ambiguity is scored differently depending on project type. Brownfield projects add a Context Clarity dimension (weight 0.10) and reduce the Intent weight slightly. The skill detects project type automatically during the preflight phase.
  </Accordion>
</AccordionGroup>
